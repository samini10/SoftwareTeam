**This file is your WORKFLOW GUIDE.** 

** What is the WORKFLOW used?
Workflow used here is about completing user's "task or new feature request or any new project" using a structured approach to software development using specialized AI agents. Each of the AI agents MUST follow the " WORKFLOW GUIDE as well as their role specific agent files" in [`ai-assistants/agents/`](./ai-assistants/agents/) folder.  The Specialied agents with their agent files as well as their main tasks are as follows:

'Product Owner agent' with agent file [`ai-assistants/agents/product-owner-agent.md`](./ai-assistants/agents/product-owner-agent.md): Customer-facing, gathers requirements, creates user stories documents, clarifies acceptance criteria, etc
'Architect agent' with agent file [`ai-assistants/agents/architect-agent.md`](./ai-assistants/agents/architect-agent.md): Designs systems, creates technical specifications and interfaces and capture everything in documents.
'Developer agent' with agent file [`ai-assistants/agents/developer-agent.md`](./ai-assistants/agents/developer-agent.md): Implements features and writes code
'Tester agent' with agent file [`ai-assistants/agents/tester-agent.md`](./ai-assistants/agents/tester-agent.md): Tests and validates implementations and captures test results, test reports and issues in documents.
'IT agent' with agent file [`ai-assistants/agents/it-agent.md`](./ai-assistants/agents/it-agent.md): Manages infrastructure and releases
'Cost Analyst agent with agent file [`ai-assistants/agents/cost-analyst-agent.md`](./ai-assistants/agents/cost-analyst-agent.md): Estimates token costs, captures in documents, warns before expensive operations.

You will act as these specialied agents. ## Git Workflow

All work in this project is organized using Git branches at two levels:

1. **Task level:** When the user gives you new work — whether it is building a new project, adding a feature, fixing a bug, or any other request — a task branch `master_{task_name}` is created from the `main` branch in the user's forked repository. This task branch is the single destination for all work related to that request.

2. **Agent level:** As each agent (Architect, Developer, Tester, etc.) works on their part of the task, they create their own branch from the task branch. When an agent completes their work, their branch is merged back into the task branch — either directly or through a pull request if the user requests a review.

### How branches work

```
main                            ← template branch (never modified directly)
  └── master_{task_name}                                 ← task branch (one per user request)
        ├── aider/product-owner-{task_name}-{sessionID}     ← Product Owner's work → merges back to task branch
        ├── aider/cost-analyst-{task_name}-{sessionID}      ← Cost Analyst's work → merges back to task branch
        ├── aider/architect-{task_name}-{sessionID}         ← Architect's work → merges back to task branch
        ├── aider/it-{task_name}-{sessionID}                ← IT's work (setup) → merges back to task branch
        ├── aider/developer-{task_name}-{sessionID}         ← Developer's work → merges back to task branch
        ├── aider/tester-{task_name}-{sessionID}            ← Tester's work → merges back to task branch
        └── aider/it-release-{task_name}-{sessionID}        ← IT's work (release) → merges back to task branch
```

### Branch naming

- **Template branch:** `main`
- **Task branch:** `master_{task_name}` (e.g., `master_login-page`)
- **Agent branch:** `aider/{agent}-{task_name}-{sessionID}` (e.g., `aider/developer-login-page-abc123`)

### Rules

- Every new user request gets a new task branch created from `main`
- All agent branches are created from the task branch, not from the template branch
- All pull requests target the task branch `master_{task_name}`, never main or master
- Iterations, fixes, and improvements related to the same user request all stay within the same task branch

### Creating a Task Branch (Product Owner does this at Step 2)

```bash
git checkout main
git checkout -b master_{task_name}
git push -u origin master_{task_name}
```

### Creating an Agent Branch

```bash
git checkout master_{task_name}
git checkout -b aider/{agent}-{task_name}-{sessionID}
```

### Creating a PR

```bash
gh pr create --base master_{task_name} \
  --head aider/{agent}-{task_name}-{sessionID} \
  --title "[Agent Name] Work description" \
  --body "## Summary
[What was accomplished]

## Ready for
[Next Step Name]"
```

### If PR Creation Fails

1. `gh auth status` — check authentication
2. `gh repo set-default OWNER/REPO` — set repository context
3. `git remote -v` — verify remotes point to your fork

### Merging Before Handover

When the user declines a PR and asks to hand over directly, you MUST merge your agent branch into the task branch first. This ensures the next agent has all previous work. Also, always delete the agent branch after merging to keep the repository clean:

```bash
git checkout master_{task_name}
git merge aider/{agent}-{task_name}-{sessionID}
git push origin master_{task_name}
git branch -d aider/{agent}-{task_name}-{sessionID}
git push origin --delete aider/{agent}-{task_name}-{sessionID}
```

If you skip this step, the next agent will branch from an outdated task branch and miss all your work.
